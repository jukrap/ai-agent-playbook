# 깊은 모듈 설계 검토

코드베이스가 지나치게 조각나 보이거나, 보조 함수 또는 모듈이 내부를 너무 많이 드러내거나, 제안된 리팩터링이 복잡도를 새 위치로 옮기기만 할 수 있을 때 이 참고 자료를 사용합니다.

## 얕은 경계 찾기

다음과 같은 모듈을 찾습니다.

- 공개 인터페이스가 숨기려는 구현만큼 복잡합니다.
- 호출자가 순서, 내부 플래그, 캐시 세부사항, 저장 방식, 오류 처리 흐름을 알아야 합니다.
- 하나의 개념을 이해하려면 여러 작은 파일을 열어야 합니다.
- 보조 함수가 주로 테스트 편의 때문에 존재하지만 실제 판단을 집중시키지 않습니다.
- 공유 유틸리티가 서로 관련 없는 호출자를 많이 갖고 도메인 소유자가 분명하지 않습니다.
- 래퍼가 호출자가 알아야 할 지식을 줄이지 않고 바탕 API 이름만 바꿉니다.

## 삭제 테스트

분리 또는 추출을 제안하기 전에 묻습니다.

- 이 모듈이 사라지면 복잡도가 더 집중되는가, 아니면 호출자로 이동할 뿐인가?
- 대체 인터페이스는 뒤의 구현보다 더 적은 개념을 갖는가?
- 어떤 불변식을 한 곳에서 더 쉽게 강제할 수 있는가?
- 공개 동작이 안정적이라서 어떤 기존 테스트가 그대로 살아남는가?
- 무엇이 바꾸기 더 어려워지고, 그것이 허용 가능한가?

답이 대부분 "복잡도가 이동한다"라면 아키텍처 개선이라고 부르지 않습니다.

## 두 번 설계하기

중요한 경계 재설계는 선택 전에 적어도 두 개 이상의 실질적으로 다른 인터페이스를 비교합니다.

- 최소 인터페이스: 진입점 1~3개, 호출당 최대 효과.
- 유연한 인터페이스: 더 많은 확장 지점, 명시적 정책/설정 객체.
- 호출자 최적화 인터페이스: 가장 흔한 사용 사례가 사소하고 안전합니다.
- 포트/어댑터 인터페이스: 외부 의존성이 명확한 어댑터 뒤에 있습니다.

대안을 다음 기준으로 비교합니다.

- 깊이: 인터페이스가 얼마나 많은 구현 복잡도를 숨기는가.
- 국소성: 미래 변경이 얼마나 한 모듈 안에 머무는가.
- 결합도: 어떤 의존성이 경계를 건너는가.
- 테스트 가능성: 내부를 드러내지 않고 어떤 동작을 검증할 수 있는가.
- 이동 비용: 어댑터, 호환성 보조 계층, 단계적 import 이동.

## 수집할 증거

- import 그래프 또는 패키지 의존 방향.
- 공개 export, 호출자, 호출 위치.
- 반복되는 매개변수 묶음, 옵션 객체, 플래그, 오류 처리 패턴.
- 문서, 테스트, UI 라벨, 데이터베이스 이름, API 이름에서 나온 도메인 용어.
- 원하는 안정 동작을 표현하는 기존 테스트.

## 산출물

권고는 간결하게 유지합니다.

- 현재 마찰: 변경이나 이해를 비싸게 만드는 요소.
- 더 깊은 인터페이스 후보: 복잡도를 숨길 수 있는 가장 작은 안정 계약.
- 경계 뒤로 이동하는 것: 파싱, 검증, 저장, 캐시, 외부 호출, 정책, 조율.
- 이동 단위: 첫 안전한 단계와 호환성 계획.
- 검증: 테스트, import 점검, 그래프 점검, 동작 근거.

## 중단 조건

- 호출자 또는 import 근거를 확인하지 않았습니다.
- 재설계가 전체 아키텍처를 한 번에 교체하는 데 의존합니다.
- 새 인터페이스가 제거하는 개념보다 더 많은 개념을 도입합니다.
- 도메인 질문이 아직 불명확해서 모듈이 얕아 보이는 것뿐입니다.
- 사용자가 좁은 수정을 요청했고 경계 재설계가 안전을 위해 필요하지 않습니다.
