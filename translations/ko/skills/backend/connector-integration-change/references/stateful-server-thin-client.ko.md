# 상태를 가진 서버와 얇은 클라이언트

커넥터 또는 외부 연동이 로컬 클라이언트, SDK, CLI 명령, 작업자, 큐, 캐시, 색인, 서버 측 자격 증명 해석을 가질 때 이 참고 자료를 사용합니다.

## 경계 규칙

- 얇은 클라이언트는 명령 파싱, 서버 위치 확인, 입력 포장, API 호출, 결과 표시만 맡습니다.
- 상태를 가진 서버는 설정, 자격 증명, 커넥터 등록, 작업, 큐, 캐시, 색인, 지속 메타데이터를 소유합니다.
- 클라이언트가 로컬 바이트를 업로드해야 한다면 이를 특수 전송 단계로 격리하고, 업로드 이후 커넥터 동기화 계약은 일관되게 유지합니다.
- 에이전트용 스킬이나 CLI 래퍼가 커넥터 상태의 숨은 기준점이 되게 하지 않습니다.

## Connector 모델

다음 개념을 분리해서 추적합니다.

- 커넥터: 등록된 출처 또는 외부 서비스.
- 객체: 커넥터 아래에서 주소를 지정할 수 있는 항목. 예: 파일, 테이블, 이슈, 메시지, 레코드.
- 커넥터 작업: 전체 커넥터에 대한 동기화 또는 가져오기 실행.
- 객체 과업: 커넥터 작업이 만든 하나의 작업 단위.
- 큐: 지속되는 작업 점유와 재시도 상태.
- 캐시: 파생 변환 결과 또는 비용이 큰 중간 산출물.
- 색인: 검색하거나 질의할 수 있는 파생 데이터.
- 메타데이터 저장소: 등록, 상태, 작업/과업 상태, 지문, 감사 기록.

## 자격 증명 해석

- 메타데이터에는 원문 비밀값보다 `env:NAME` 또는 `file:/path` 같은 참조를 선호합니다.
- 참조는 작업자와 커넥터가 실행되는 서버 환경에서 해석합니다.
- 커넥터 계약에 필요한 권한 범위와 권한 확대를 기록합니다.
- 누락, 만료, 폐기, 권한 부족, 형식 오류, 잘못된 환경의 자격 증명 사례를 검증합니다.
- 자격 증명 값을 로그, 테스트 고정 데이터, 스냅샷, 작업 기록, 생성 문서, MCP 응답에 넣지 않습니다.

## 작업과 과업 안정성

- 커넥터 작업은 상태, 시작/완료 시각, 실패 이유, 재시도 횟수, 영향받은 객체 수를 노출해야 합니다.
- 객체 과업은 멱등성을 갖거나 재실행/복구 전략을 가져야 합니다.
- 작업자에는 시간 초과, 취소, 종료, 독성 메시지, 중복 전달 처리 동작이 필요합니다.
- 계약이 명시적으로 부분 성공을 지원하지 않는 한 부분 실패를 전체 커넥터 성공처럼 보이게 하면 안 됩니다.
- 캐시와 색인 무효화는 출처 지문 또는 버전이 있는 변환 입력에 연결합니다.

## API와 클라이언트 계약

- API는 사용자가 무엇이 등록됐고, 실행 중이고, 실패했고, 검색 가능하고, 복구가 필요한지 답할 수 있을 만큼의 상태를 노출해야 합니다.
- SDK와 CLI는 서버에 있는 업무 규칙을 중복 구현하지 않습니다.
- 오류는 인증, 권한, 검증, 호출 제한, 원격 장애, 재시도 가능한 내부 실패, 영구적으로 지원하지 않는 작업을 구분합니다.
- 검색/읽기 연결은 서버 측 출처 범위를 강제하고 다시 열 수 있는 위치 정보를 반환합니다.

## 검증

- 커넥터 등록/수정/삭제 계약 테스트.
- 서버 환경에서 자격 증명 참조 해석 테스트.
- 커넥터 작업 생명주기 테스트: 대기, 실행 중, 부분 실패, 성공, 재시도, 취소.
- 객체 과업 멱등성과 중복 전달 테스트.
- 출처 변경, 권한 변경, 스키마 변경 뒤 캐시/색인 무효화 테스트.
- 새 서버에 대해 클라이언트가 상태 없이 복구되는지 확인하는 CLI/SDK 스모크 테스트.
