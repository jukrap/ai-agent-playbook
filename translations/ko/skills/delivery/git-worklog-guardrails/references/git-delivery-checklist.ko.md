# Git delivery 점검표

파일 스테이징, 커밋, 푸시, PR 문구 작성, 작업 기록 작성 전에 사용합니다.

## 사전 점검

- 현재 브랜치, 추적 브랜치, 원격 저장소, 변경 파일, 스테이징된 파일, 무시된 로컬 전용 폴더를 확인합니다.
- 저장소 커밋/PR 지침이 있으면 읽습니다.
- 어떤 변경이 내가 만든 것이고 어떤 변경이 관련 없는 사용자 변경인지 확인합니다.
- 저장소 스크립트나 문서에서 검증 명령을 확인합니다.
- push 또는 PR 생성이 명시적으로 요청되었는지 확인합니다.

## 스테이징

- 작업과 관련된 명시적 경로만 스테이징합니다.
- 로컬 전용 작업공간, 생성된 임시 산출물, reference audit note, credential, private environment file을 스테이징하지 않습니다.
- 커밋 전에 스테이징된 이름과 요약을 확인합니다.
- 건드린 파일에 관련 없는 편집이 있으면 diff를 조심스럽게 검토하고 가능하면 의도한 hunk만 스테이징합니다.

## 커밋 메시지

- 저장소 커밋 규칙을 사용합니다.
- 제목은 브랜치 이름이나 채팅 제목이 아니라 핵심 결과를 기준으로 씁니다.
- diff가 여러 관심사를 포함하거나 미래 독자에게 범위/위험 맥락이 필요하면 본문을 포함합니다.
- 실제 실행한 검증만 검증 섹션에 씁니다.
- agent, model, generated-by, co-author, signature trailer를 추가하지 않습니다.

## Push와 PR

- 의도한 브랜치만 의도한 원격 저장소로 push합니다.
- 명시적 지시 없이 보호 브랜치나 공유 브랜치로 push하지 않습니다.
- PR 문구는 실제 diff와 실제 검증을 기반으로 씁니다.
- 공유 동작, 설정, 패키징, 운영 변경에는 되돌림 메모를 포함합니다.
- 실제 UI 변경을 확인했을 때만 스크린샷을 포함합니다.

## 작업 기록

마일스톤 종료, 막힌 지점 해결, 방향 전환, 긴 디버깅, 남은 운영 위험이 있으면 작업 기록을 씁니다. 오래 남을 사실은 커밋 메시지에만 두지 말고 playbook memory나 프로젝트 문서에 둡니다.
