# 에이전트 스킬 작성

에이전트가 찾기 쉽고 오용하기 어려운 작은 재사용 스킬을 만듭니다.

## 작업 흐름

1. 해당 동작이 여러 프로젝트에서 재사용 가능한지 확인합니다. 프로젝트별 상시 정책이라면 스킬 대신 템플릿이나 프로젝트 문서에 둡니다.
2. 트리거를 먼저 씁니다. `description`은 `Use when...`으로 시작하고, 작업 절차가 아니라 언제 스킬을 불러와야 하는지 설명합니다.
3. `SKILL.md`는 간결하게 유지합니다. 트리거, 핵심 절차, 참고 문서를 읽어야 하는 경우만 둡니다.
4. 긴 규칙, 예시, API 세부사항, 체크리스트는 한 단계 아래의 `references/*.md` 파일로 옮깁니다.
5. 검증, 포맷팅, 산출물 생성처럼 결과가 결정적인 반복 작업에만 스크립트를 추가합니다.
6. 현실적인 요청과 헷갈릴 수 있는 요청 두세 개로 트리거를 시험하고, 너무 자주 불러와질 것 같으면 문구를 좁힙니다.
7. 다른 스킬을 추가하기 전에 기존 스킬과 프로젝트 지침과의 충돌을 확인합니다.
8. 저장소 유지보수 흐름에 따라 색인, 번역, 검증 메타데이터, 설치된 복사본을 갱신합니다.

## 경계

- 좋은 규칙이라고 해서 모두 스킬로 만들지 않습니다.
- 제품 범위, 마일스톤 상태, 브랜치 정책, 개인 경로, 특정 기기 설정을 설치형 스킬에 넣지 않습니다.
- 프로젝트 `AGENTS.md`, 로컬 문서, 기존 작업 흐름 스킬 묶음을 중복하는 긴 스킬을 만들지 않습니다.
- 외부 스킬 묶음을 통째로 들여오기보다 짧은 로컬 규칙이나 이름을 분명히 바꾼 집중 스킬이 더 명확한지 먼저 봅니다.
- 특정 에이전트 전용 훅, 슬래시 명령, 환경 변수에 의존하지 않습니다.

## 빠른 구조

```text
skills/<category>/<skill-name>/
  SKILL.md
  references/<detail>.md   # 세부사항이 너무 길거나 드물게 필요할 때만
  scripts/<tool>           # 결정적인 재사용 작업일 때만
  agents/openai.yaml       # UI 메타데이터가 유용할 때만
```

## 품질 체크리스트

- 스킬 이름은 소문자와 하이픈으로 씁니다.
- Frontmatter에는 `name`, `description`만 있습니다.
- Description은 `Use when...`으로 시작합니다.
- 본문은 빠르게 훑을 수 있을 만큼 짧습니다.
- 트리거는 더 넓은 작업 흐름 스킬과 충돌하지 않을 만큼 좁습니다.
- 참고 문서는 한 단계 아래에만 둡니다.
- 영어 원본과 한국어 번역을 함께 갱신합니다.
- 설치된 복사본을 동기화하기 전에 검증 명령이 통과합니다.
