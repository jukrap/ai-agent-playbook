# Git과 작업 기록 체크리스트

파일 스테이징, 커밋 메시지 작성, 푸시, PR 생성, 작업 기록 작성 전에 이 참고 문서를 사용합니다.

프로젝트가 `templates/agents/global/GIT.md`를 복사했다면 짧은 루트 정책과 이 상세 가이드를 서로 맞춰 유지합니다. 루트 파일은 상시 규칙용이고, 이 참고 문서는 전체 절차 체크리스트용입니다.

## 스테이징 전

```bash
git status --short --branch
git remote -v
git branch --show-current
```

관련 없는 dirty file, 현재 브랜치, upstream, remote, 보호 브랜치, local-only 정책을 확인합니다.

## 스테이징

- `git add .`, `git add -A`보다 명시적 경로 스테이징을 선호합니다.
- 작업과 관련된 파일만 stage합니다.
- 프로젝트가 명시적으로 추적하지 않는 한 local-only 문서, 생성 산출물, 작업 기록, 프롬프트 초안, 참고 자산을 stage하지 않습니다.
- 스테이징 후 파일 이름과 요약을 모두 확인합니다.

```bash
git diff --cached --name-only
git diff --cached --stat
```

## 검증

- 저장소가 정의한 script만 사용합니다.
- 있을 때 일반적인 순서는 lint, tests, build입니다.
- 실행하지 않은 명령이 통과했다고 말하지 않습니다.
- 명령이 실패하면 원인을 읽고, 관련 파일만 고친 뒤, 같은 명령을 다시 실행하고 실제 결과를 보고합니다.

## 커밋 메시지 정책

목표:

- 요청된 커밋 전략에 따라 현재 diff 또는 staged diff를 기준으로 최종 커밋 메시지를 1개 또는 여러 개 작성합니다.
- 사람이 빠르게 읽고 변경 의도와 핵심 결과를 파악할 수 있게 합니다.
- 변경이 논리적으로 분리되어 있고 사용자가 커밋 또는 커밋 준비를 요청했다면 별도 커밋을 선호합니다.
- 사용자가 단일 커밋을 요청했거나 저장소 규칙이 작업당 커밋 1개를 선호한다면 가장 중심적인 논리 변경을 제목에 쓰고 부수 변경은 본문에 둡니다.

출력:

- 요청된 최종 커밋 메시지 또는 커밋 계획만 출력합니다.
- 설명, 후보 목록, 이유 설명, 따옴표, code fence를 붙이지 않습니다.
- 필요한 구조만 씁니다: 제목, 제목과 본문, 제목과 본문과 검증, 제목과 본문과 검증과 footer.

형식:

```text
type(scope): summary
```

- Conventional Commit type은 `feat`, `fix`, `design`, `style`, `refactor`, `perf`, `test`, `docs`, `build`, `ci`, `chore`를 사용합니다.
- scope는 유용하고 저장소 규칙과 맞을 때만 씁니다.
- subject는 저장소의 언어 규칙을 따릅니다. 확인된 규칙이 없으면 subject와 body는 사용자 또는 팀의 작업 언어에 맞춥니다. Conventional Commit type과 scope는 저장소가 다른 형식을 쓰는 것이 확인되지 않는 한 영어로 유지합니다.

제목 규칙:

- 한 줄로 쓰고 끝에 마침표를 붙이지 않습니다.
- 과정이 아니라 구체적인 결과를 씁니다.
- subject는 사용자, 팀, 저장소의 주 사용 언어로 씁니다.
- 브랜치명, 채팅 제목, 작업 제목을 그대로 복사하지 않습니다.
- `fix: update`, `chore: changes`, `refactor: cleanup`처럼 막연한 제목을 피합니다.
- `작업`, `처리`, `수정`, `변경`, `개선` 같은 일반 단어를 단독 핵심어처럼 쓰지 않습니다.

본문 규칙:

- 단순 변경이면 본문을 생략합니다.
- 변경에 나중에 참고할 만한 context가 있으면 `- ` 불릿 2~6개를 씁니다. 범위가 좁은 변경은 불릿 1개도 괜찮습니다.
- diff에서 확인되는 무엇을 바꿨는지, 왜 중요한지, 영향 범위, 주의점을 보존합니다.
- 불릿은 결과 중심으로 쓰고 실제 diff에 근거합니다.
- 구현 과정을 장황하게 설명하지 않습니다.

검증 규칙:

- 실제로 실행한 확인에 대해서만 `Verification` 또는 `검증` 섹션을 넣습니다.
- 명령 이름이나 수동 확인 시나리오를 짧게 적습니다.
- 실행하지 않은 test, browser, device, build, deployment, scenario check를 언급하지 않습니다.

이슈/footer 규칙:

- 이슈 번호를 확실히 아는 경우에만 참조를 넣습니다.
- 저장소 규칙이 다르지 않으면 `Refs #123`, `Closes #123`를 우선합니다.
- 같은 이슈를 제목과 footer에 중복하지 않습니다.

금지:

- 명시적으로 요구되지 않았다면 에이전트, 모델, generated-by, co-authored-by, signed-off-by, 이메일 서명을 추가하지 않습니다.
- 변경, 테스트, 이슈 번호, 검증 결과를 지어내지 않습니다.

## Push

push 전:

```bash
git status --short --branch
git log --oneline --decorate -5
git remote -v
```

- 현재 브랜치와 대상 remote를 확인합니다.
- fork/origin/upstream 관계를 확인합니다.
- 명시적 지시 없이 보호 브랜치, 배포 브랜치, 공유 브랜치에 푸시하지 않습니다.
- local-only 파일이나 검증되지 않은 작업을 푸시하지 않습니다.

## PR 본문 정책

- 저장소 템플릿이 있으면 따릅니다.
- 템플릿이 없으면 요약, 관련 이슈, 변경 사항, 리스크, 테스트/검증, 롤백 플랜, 스크린샷/영상을 포함합니다.
- PR 문장은 저장소의 언어 규칙을 따릅니다. 확인된 규칙이 없으면 사용자 또는 팀의 작업 언어에 맞춥니다.
- 무엇이 달라졌는지를 먼저 쓰고, 왜 바꿨는지를 그 다음에 씁니다.
- 커밋 순서가 아니라 리뷰 관점 기준으로 변경을 묶습니다.
- 리스크는 `Low`, `Medium`, `High` 중 하나로 정하고 실제 영향 기준의 이유를 붙입니다.
- 실제 수행한 확인만 적습니다.
- 공용 동작, 설정, 운영 영향 가능성이 있는 변경에는 롤백 절차를 적습니다.
- UI 변경일 때만 스크린샷/영상을 포함합니다.
- raw diff, commit log, 지어낸 이슈 번호, 검증하지 않은 테스트 결과를 붙이지 않습니다.

## 작업 기록 작성 기준

다음 경우 작업 기록을 작성합니다.

- 마일스톤 완료
- blocker 또는 반복 실패
- 큰 방향 변경
- 원인 분석이 유용한 긴 디버깅
- 배포, 네이티브, 출력, 권한, API 계약 변경

## 작업 기록 언어와 형태

- 작업 기록 언어는 대상 시스템과 팀 언어에 맞춥니다.
- 한국어 Jira 프로젝트 또는 사용자가 한국어로 요청한 상황이면 한국어 섹션 제목과 한국어 문장을 사용합니다.
- 영어 중심 티켓, 저장소, 이해관계자가 대상이면 영어 섹션 제목과 영어 문장을 사용합니다.
- 명령, 패키지명, API명, 기술 식별자는 원문을 유지합니다.

한국어 Jira worklog는 아래 형태를 우선합니다.

```md
### 제목

`작업 결과 중심 제목`

### 요약

-

### 배경

-

### 확인된 문제

-

### 원인

-

### 적용 내용

-

### 검증

-

### 잔여 확인 사항

-
```

작성 원칙:

- 파일명, 경로, 특정 마크다운 문서 이름을 그대로 나열하지 않습니다.
- 다른 사람이 별도 자료 없이 읽어도 이해되도록 변경 이유와 결과를 설명식으로 적습니다.
- 특정 사람이나 외부 자료명을 내세우기보다 실제로 어떤 기준과 판단으로 바꿨는지 적습니다.
- “무엇을 참고했다”보다 “무엇이 문제였고 어떻게 정리했는지”를 우선합니다.

커밋 메시지는 다음 독자를 위한 간결한 context를 보존할 수 있습니다. 더 깊은 reasoning, blocker, decision history, confirmed cause, verification detail, remaining risk는 작업 기록에 둡니다.

프로젝트가 작업 기록을 미래의 에이전트나 유지보수자를 위한 지속적 context로 사용한다면, 작업 기록을 커밋 메시지 수준의 짧은 요약으로 줄이지 않습니다.
