# 작업 표면 UI 폴리시 점검

제품 방향이나 디자인 시스템을 다시 쓰지 않고 보이는 UI를 다듬을 때 이 참고 자료를 사용합니다.

## 작업에서 시작하기

- 화면이 지원해야 하는 주요 작업과 사용자가 가장 먼저 할 수 있어야 하는 동작을 식별합니다.
- 요청이 재설계를 요구하지 않는 한 기존 내비게이션, 정보 구조, 컴포넌트 관례를 보존합니다.
- 보조 장식보다 현재 작업 표면을 먼저 개선합니다.
- 운영 도구에서는 훑기 쉬운 밀도, 안정적인 컨트롤, 명확한 상태, 필터, 표, 폼, 감사 단서를 우선합니다.
- 포트폴리오, 미디어, 게임, 편집물 화면에서는 표현 요소를 실제 콘텐츠나 상호작용과 연결합니다.

## 레이아웃과 계층

- 첫 가시 영역을 단순 장식이 아니라 작업에 바로 유용하게 만듭니다.
- 컨트롤, 도구 모음, 카운터, 격자, 보드, 카드, 미디어 프레임, 고정 형식 요소에는 안정적인 크기를 둡니다.
- 지원 화면 너비에서 텍스트, 아이콘, 라벨, 컨트롤, 동적 콘텐츠가 겹치지 않게 합니다.
- 실무형 화면에서는 중첩 카드, 장식 래퍼, 불필요한 섹션 장식을 피합니다.
- 글자 크기는 컨테이너에 맞춥니다. 작은 패널에는 큰 히어로 제목이 아니라 작은 제목이 필요합니다.

## 상태 범위

- 컴포넌트가 해당 상태를 노출한다면 로딩, 빈 상태, 오류, 비활성, 선택, hover, focus, active, 성공, 파괴적 동작, 긴 콘텐츠 상태를 확인합니다.
- 파괴적이거나 되돌리기 어려운 동작은 일반 동작과 시각적으로 구분합니다.
- 프로젝트에 이미 있는 키보드 포커스, 스크린리더 라벨, 보이는 포커스 링, 비활성 의미를 보존합니다.
- 검증 메시지, 토스트, 대화상자, 드롭다운, 고정 영역이 서로를 가리지 않는지 확인합니다.

## 반응형과 콘텐츠 스트레스

- 가능하면 지원하는 가장 작은 모바일 너비, 일반 모바일, 노트북, 데스크톱 너비를 확인합니다.
- 긴 라벨, 번역 문구, 큰 숫자, 빈 데이터, 많은 항목, 좁은 컬럼, 검증 오류로 스트레스 테스트합니다.
- viewport 기준 글자 크기 조정보다는 줄바꿈, 접근 가능한 전체 텍스트가 있는 말줄임, 표 가로 스크롤, 대체 레이아웃을 선호합니다.
- 고정/sticky 요소가 포커스 대상, 브라우저 UI, 하단 동작, 주요 콘텐츠를 가리지 않는지 검증합니다.

## 시각 시스템 적합성

- 일회성 스타일을 추가하기 전에 공유 컴포넌트, 토큰, variant, class 관례, 간격 체계를 재사용합니다.
- 단일 색상 계열 팔레트, 임의 그림자, 과한 gradient, 작업을 명확히 하지 않는 장식 배경을 피합니다.
- 익숙한 명령에는 텍스트만 있는 pill보다 기호가 더 명확할 때 아이콘을 사용합니다.
- 디자인 시스템이 분명하지 않다면 주변 컴포넌트를 따르고 폴리시 범위를 국소적으로 유지합니다.

## 검증

- 프로젝트를 로컬에서 실행할 수 있으면 변경 화면을 브라우저에서 렌더링합니다.
- 실제로 영향을 받은 너비와 상태를 캡처하거나 검사합니다.
- 새 컨트롤의 키보드 포커스 순서를 확인합니다.
- 관련되는 경우 프로젝트가 정한 lint, type, test, build 명령을 실행합니다.
- 실제로 확인한 상태, 너비, 명령만 보고합니다.

## 중단 조건

- 지원 화면 너비에서 텍스트나 컨트롤이 겹칩니다.
- 사용자가 주요 동작에 도달하거나 이해할 수 없습니다.
- 디자인 방향 요청 없이 새 시각 언어를 도입합니다.
- 이상적인 샘플 콘텐츠에서만 UI를 확인했습니다.
- 접근성 상태가 제거되거나 덜 보이게 바뀝니다.
