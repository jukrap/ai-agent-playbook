# 레거시 변경 통제

현재 동작, 운영 절차, downstream 계약을 보존해야 하는 레거시 변경에 사용합니다.

## 변경 순서

1. 수정 전에 현재 동작을 확보합니다.
2. 반드시 바뀌어야 하는 가장 작은 사용자 관찰 동작을 식별합니다.
3. 그 동작을 만드는 실행 경로를 추적합니다.
4. 인접 경로, 템플릿, 스크립트, 작업, 데이터베이스 산출물, 생성 산출물, 배포 파일을 검색합니다.
5. 요청을 만족하는 가장 좁은 변경을 만듭니다.
6. 유지되어야 하는 동작과 바뀐 동작을 따로 검증합니다.
7. 되돌림 경로, 남은 위험, 검증하지 못한 의존성을 기록합니다.

## 수집할 근거

- 활성 진입점, 경로, 명령, 스케줄러 또는 수동 운영 절차.
- 빌드 산출물이 있으면 원본 파일과 실행 산출물.
- 변경 전후 데이터 형태.
- 바뀐 필드, 선택자, 경로, 파일, 프로시저 시그니처의 호출자 또는 소비자 목록.
- 검증에 사용한 명령, 화면 흐름, 샘플 파일 또는 쿼리.

## 변경 경계

- 공유 전역 경로를 바꾸기보다 좁은 어댑터나 분기를 선호합니다.
- 기능 작업 중에는 이름과 형식을 정리하기보다 보존하는 쪽을 선호합니다.
- 큰 재작성보다 특성화 테스트, 캡처한 fixture, 수동 시나리오를 선호합니다.
- 빠진 백엔드, 데이터베이스, 브라우저, 장치 계약을 조용히 가정하지 말고 명시적인 막힌 지점으로 둡니다.

## 검증 매트릭스

- 기존 경로: 대표 시나리오 하나 이상으로 이전 동작이 유지되는지 증명합니다.
- 새 경로: 변경된 시나리오에서 요청 동작이 작동하는지 증명합니다.
- 실패 경로: 검증, 오류 표시, 재시도 또는 되돌림이 예측 가능하게 동작하는지 증명합니다.
- 운영 경로: 건드렸다면 빌드, 패키징, 배치, 프린터, 배포 또는 수동 runbook 동작을 증명합니다.

## 위험 신호

- 파일이 미사용처럼 보이지만 배포 산출물, 스크립트 include, 실행 설정, 예약 작업, 생성 manifest에 나타납니다.
- 같은 필드 이름이 UI, 서버 바인딩, SQL, 내보내기, 가져오기, 보고서 코드에 나타납니다.
- 수동 운영 절차가 유일한 복구 경로입니다.
- 테스트 환경이 위험을 소유한 브라우저, 데이터베이스, 프린터, 스케줄러, 파트너 연동을 실행할 수 없습니다.
