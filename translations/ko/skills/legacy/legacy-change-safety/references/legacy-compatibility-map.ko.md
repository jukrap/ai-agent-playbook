# 레거시 호환성 지도

레거시 시스템을 바꾸기 전에 숨은 결합을 파악할 때 사용합니다.

## 지도 축

- 실행: 진입점, 활성 경로, 예약 작업, 브라우저 모드, 장치 플러그인, 서버 모듈.
- 데이터: 테이블, 뷰, 저장 프로시저, 트리거, 파일, 내보내기, 가져오기, 보고서 쿼리.
- UI: 템플릿, 부분 템플릿, 공유 선택자, 전역 스크립트, 인라인 handler, 레이아웃 include.
- 빌드: 생성 산출물, 복사되는 정적 파일, 번들링 단계, 패키징 규칙, 배포 설정.
- 운영: cron 일정, Windows 작업, 수동 실행, 되돌림 스크립트, 백업, 지원 절차.
- 외부 계약: 파트너 파일, API 소비자, downstream 보고서, 프린터 경로, 브라우저 정책, SSO/세션 가정.

## 확인 질문

- 어떤 산출물을 수정하고, 어떤 산출물이 실제로 실행되는가?
- 어떤 코드 경로가 값을 쓰고, 어떤 코드 경로가 나중에 읽는가?
- 어떤 소비자가 정확한 이름, 순서, 형식, 인코딩, 시각에 의존하는가?
- 어떤 환경별 규칙이 해당 경로의 활성 여부를 결정하는가?
- 어떤 수동 프로세스가 형태나 시각 변화 때문에 실패하는가?

## 검색 단서

- 필드 이름과 라벨.
- CSS 선택자와 데이터 속성.
- 경로 이름, 액션 이름, 명령 이름, 스케줄러 이름.
- 파일 확장자, 폴더 이름, 파일명 패턴.
- 저장 프로시저 이름, 트리거 이름, 테이블 이름, 내보내기 컬럼 이름.
- 설정 키, 환경 변수, 레지스트리 키, 그룹 정책 이름.

## 산출물

위험이 작지 않으면 짧은 호환성 메모를 남깁니다.

- 변경된 계약.
- 확인한 소비자.
- 알 수 없는 소비자.
- 검증 근거.
- 되돌림 또는 격리 계획.

도구가 만든 지도는 사람이나 검토자가 실제 운영 시스템과 일치한다고 확인하기 전까지 장기 기억으로 승격하지 않습니다.
