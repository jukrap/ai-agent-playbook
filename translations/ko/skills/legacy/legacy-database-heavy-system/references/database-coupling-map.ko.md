# 데이터베이스 결합 지도

데이터베이스 산출물이 업무 동작을 포함하는 레거시 시스템을 바꾸기 전에 사용합니다.

## 추적할 산출물

- 테이블, 컬럼, 인덱스, 제약, sequence, lookup row.
- 뷰, materialized view, synonym, linked-server 참조.
- 저장 프로시저, 함수, package, 트리거, 예약 작업.
- ORM 매핑, 직접 SQL 문자열, 보고서 쿼리, 내보내기 쿼리, ETL 작업.
- 마이그레이션 스크립트, seed 스크립트, 백업/복원 스크립트, 수동 지원 쿼리.

## 결합 질문

- 어떤 애플리케이션 경로가 이 산출물을 호출하는가?
- 어떤 보고서, 내보내기, 가져오기, 대시보드, 파트너 파일이 같은 형태에 의존하는가?
- 트리거나 작업이 애플리케이션 코드가 쓴 뒤 데이터를 바꾸는가?
- 프로시저가 여러 결과 집합이나 overload된 형태를 반환하는가?
- 변경이 잠금, 격리, deadlock 위험, 배치 실행 시간에 영향을 주는가?
- 데이터 손실 없이 되돌릴 수 있는가?

## 변경 점검

- 프로시저 매개변수: 이름, 순서, 타입, null 허용, 기본값, output 동작.
- 결과 집합: 컬럼 이름, 순서, 타입, scale, null 허용, row ordering.
- 트랜잭션: begin/commit 위치, rollback 동작, 재시도 안전성, 멱등성.
- 데이터 마이그레이션: backfill, 호환 기간, dual read/write 필요성, 검증 쿼리.
- 권한: 애플리케이션 role, 보고서 role, 작업 계정, 지원 계정.

## 검증

- 대표 기존 데이터.
- 경계 데이터: 관련 있으면 null, empty, duplicate, old row, high volume, malformed row.
- 알려진 모든 소비자에 대한 호출자 검증.
- 대용량 또는 잠금에 민감한 변경의 query plan 또는 실행 시간 점검.
- 되돌림 리허설 또는 문서화된 되돌림 한계.

데이터베이스가 계약이라면 애플리케이션 테스트만으로 충분한 증거라고 보지 않습니다.
