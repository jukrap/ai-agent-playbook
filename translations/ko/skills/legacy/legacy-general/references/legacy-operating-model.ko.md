# 레거시 운영 모델

레거시 저장소의 실행 흐름, 문서, 운영 관행이 불명확할 때 사용합니다.

## 활성 시스템 확정

1. 기능을 실행하는 명령, 서버, 스케줄러, 브라우저 또는 수동 프로세스를 식별합니다.
2. 그 경로에서 로드되는 파일을 식별합니다.
3. 원본 파일이 변환, 복사, 생성되거나 그대로 배포되는지 식별합니다.
4. 현재 환경의 설정 출처를 식별합니다.
5. 변경된 출력을 누가 또는 무엇이 소비하는지 식별합니다.

## 문서 선별

- 오래된 문서보다 실제 코드와 명령 출력을 더 강한 근거로 봅니다.
- 둘 다 있으면 오래된 README보다 최신 작업 기록이나 runbook을 더 강한 근거로 봅니다.
- 배포 설정, 스케줄러 설정, 지원 스크립트를 애플리케이션의 일부로 봅니다.
- 불확실한 문서는 관련 없는 작업 중 삭제하거나 다시 쓰지 말고 불확실하다고 표시합니다.

## 변경 전략

- 새 아키텍처 양식을 도입하기 전에 로컬 패턴을 맞춥니다.
- 새 동작은 확장하는 기존 동작 근처에 둡니다.
- 호환성에 민감한 변경 주변에는 명시적인 보호 조건을 둡니다.
- 리팩터링 전에 작은 특성화 테스트나 캡처한 수동 시나리오를 선호합니다.
- 현대화는 요청한 수정이 안전해진 뒤 별도 계획으로 분리합니다.

## 검증 전략

- 프로젝트 명령이 있으면 사용합니다.
- 테스트가 약하면 변경된 동작을 실행하는 가장 작은 실제 흐름으로 검증합니다.
- 회귀 신뢰를 위해 인접한 기존 흐름 하나 이상을 포함합니다.
- 사용할 수 없는 환경, 오래된 브라우저, 장치, 프린터, 데이터베이스, 파트너 시스템은 막힌 지점 또는 남은 위험으로 기록합니다.

## 피할 것

- 직접 import가 검색되지 않는다는 이유로 코드를 삭제합니다.
- 기능 작업 중 필드, 선택자, 파일, 프로시저 이름을 가독성 목적으로 바꿉니다.
- 미완성 백엔드나 데이터 작업이 완료된 것처럼 보이게 하는 숨은 fallback을 추가합니다.
- 배포 또는 패키징 규칙을 확인하기 전에 파일을 이동합니다.
