# IE/ActiveX 격리

오래된 브라우저 또는 장치 호환성이 필수 실행 제약일 때 사용합니다.

## 환경 기록

- 필요한 브라우저, IE 모드, 문서 모드, 호환성 보기, 기업 정책.
- 필요한 ActiveX control, plugin, device driver, certificate 또는 installer.
- 보안 zone, trusted site, registry key, group policy, 사용자 권한 가정.
- 관련 있으면 운영체제, bitness, Office/runtime 의존성, 장치 모델.
- 정확한 검증 환경 또는 사용할 수 없는 이유.

## 호환성 경계

- 프로젝트가 대상 모드용 변환을 이미 하지 않는 한 최신 JavaScript 문법을 피합니다.
- 대상 실행 환경이 지원하거나 검증된 polyfill이 없으면 브라우저 API를 피합니다.
- 이벤트 모델, 창/대화상자 동작, 인코딩, 파일/장치 접근 흐름을 보존합니다.
- 모든 호출자와 클라이언트 설치를 갱신하지 않는 한 object/embed 식별자와 매개변수를 안정적으로 유지합니다.
- 최신 브라우저 실행 결과를 IE 모드 동작의 근거로 쓰지 않습니다.

## 보안 점검

- control이 파일, 장치, 클립보드, 인증서 또는 로컬 네트워크 접근 권한을 갖는지 확인합니다.
- 테스트를 쉽게 하려고 trusted origin을 넓히거나 prompt를 약화하지 않습니다.
- 민감한 로컬 경로, 문서 내용, 인증서 세부사항, 장치 식별자를 로그나 문서에 남기지 않습니다.
- 운영에 필요한 수동 설치, 서명, 권한 단계를 문서화합니다.

## 검증

- 필요한 브라우저/문서 모드.
- 필요한 보안 zone과 기업 정책.
- plugin 또는 장치 흐름의 정상 경로.
- control 누락 또는 권한 거부 실패 경로.
- 있다면 인접한 비-plugin workflow 회귀 경로.

필요한 환경이 없으면 호환된다고 주장하지 말고 막힌 지점 또는 남은 위험으로 보고합니다.
