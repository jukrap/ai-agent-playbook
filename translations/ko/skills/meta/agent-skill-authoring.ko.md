# 에이전트 스킬 작성

에이전트가 찾기 쉽고 오용하기 어려운 작은 재사용 스킬을 만듭니다.

## 진행 절차

1. 동작이 여러 프로젝트에서 재사용 가능한지 확인합니다. 프로젝트 전용 상시 정책이라면 스킬이 아니라 템플릿 또는 프로젝트 문서에 둡니다.
2. 트리거를 먼저 씁니다. `description`은 `Use when...`으로 시작하고 작업 절차가 아니라 불러올 시점을 설명합니다.
3. `SKILL.md`는 트리거, 핵심 절차, 참고 자료를 읽을 시점만 담아 간결하게 유지합니다.
4. 긴 규칙, 예시, API 세부사항, 체크리스트는 한 단계의 `references/*.md` 파일로 옮깁니다.
5. 검증, 포맷팅, 산출물 생성처럼 결정적으로 반복되는 작업에만 스크립트를 추가합니다.
6. 현실적인 요청과 아슬아슬한 비적용 사례 2~3개로 트리거를 테스트합니다. 너무 자주 불릴 것 같으면 표현을 좁힙니다.
7. 새 스킬을 추가하기 전에 기존 skill, template, project guide를 보강하는 것으로 충분한지 확인합니다.
8. 새 스킬을 추가하기 전에 기존 스킬과 프로젝트 지침과 충돌하지 않는지 확인합니다.
9. 저장소 유지보수 흐름에 따라 색인, 번역, 검증 메타데이터, 설치된 복사본을 갱신합니다.

## 경계

- 모든 좋은 규칙을 스킬로 만들지 않습니다.
- 제품 범위, 마일스톤 상태, 브랜치 정책, 개인 경로, 특정 장비 설정을 설치형 스킬에 넣지 않습니다.
- 프로젝트 `AGENTS.md`, 로컬 문서, 기존 프로세스 스킬 묶음을 복제하는 긴 스킬을 만들지 않습니다.
- 외부 스킬 묶음을 통째로 들여오지 않습니다. 짧은 로컬 규칙이나 이름을 바꾼 집중 스킬이 더 명확할 때가 많습니다.
- 에이전트 비종속 스킬 원본에서 특정 에이전트 런타임의 hook, slash command, environment variable에 의존하지 않습니다.

## 기본 구조

```text
skills/<category>/<skill-name>/
  SKILL.md
  references/<detail>.md   # detail이 너무 길거나 드물게 필요할 때만
  scripts/<tool>           # deterministic reusable operations에만
  agents/openai.yaml       # UI metadata가 유용할 때만
```

## 품질 체크리스트

- 스킬 이름은 소문자와 하이픈을 사용합니다.
- Frontmatter는 `name`과 `description`만 포함합니다.
- Description은 `Use when...`으로 시작합니다.
- 스킬 본문은 빠르게 훑을 수 있을 만큼 짧습니다.
- 트리거는 넓은 프로세스 스킬과 충돌하지 않을 만큼 좁습니다.
- 참고 자료는 한 단계 깊이만 둡니다.
- 영어 원본과 한국어 번역을 함께 갱신합니다.
- 설치된 복사본을 동기화하기 전에 검증 명령이 통과합니다.
