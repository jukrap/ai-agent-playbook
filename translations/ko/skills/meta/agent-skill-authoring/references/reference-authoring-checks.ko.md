# 참고 자료 작성 점검

긴 참고 자료, 예시, 스택 세부사항, 외부 패턴 채택을 추가하기 전에 사용합니다.

## 참고 자료에 둘 것

- `SKILL.md`에 넣기에는 긴 상세 절차.
- 실패 모드와 중단 조건.
- 스택 프로필과 provider별 주의점.
- 검증 매트릭스와 근거 계약.
- 정리되고 일반화된 재사용 예시.
- 외부 자료에서 가져온 뒤 로컬 guidance로 다시 작성한 패턴.

## 제외할 것

- 개인 경로, 회사명, 고객명, 내부 URL, 자격 증명, 토큰, 브랜치 이름, PR 번호.
- 긴 upstream 원문 발췌.
- 일반화되지 않는 프로젝트 전용 magic value.
- 곧 오래될 운영 상태.
- 저장소 유지보수 흐름이나 더 높은 우선순위의 프로젝트 문서와 충돌하는 지침.

## 참고 자료 형태

- 언제 이 참고 자료를 쓰는지로 시작합니다.
- 섹션은 작업 중심으로 유지합니다.
- 넓은 글보다 체크리스트, 매트릭스, 근거 요구사항을 선호합니다.
- 예시는 짧고 정리된 형태로 둡니다.
- 참고 자료는 스킬 아래 한 단계 디렉터리에 둡니다.
- 모든 영어 markdown 파일에 맞는 한국어 번역을 추가합니다.

## 채택 점검

- source branding이 아니라 오래 쓸 수 있는 패턴을 추출합니다.
- 저장소의 taxonomy와 용어로 다시 작성합니다.
- 가능하면 vendor나 프로젝트 정체성보다 capability 언어를 사용합니다.
- 라이선스가 요구할 때만 attribution을 보존합니다.
- install/sync 동작에 영향을 주는 패턴이면 validator나 CLI check를 추가합니다.

## 중단 조건

- copied upstream context가 있어야만 참고 자료가 의미 있습니다.
- licensing 또는 redistribution 상태가 불명확합니다.
- 재사용 스킬 guidance보다 로컬 프로젝트 memory가 더 적합합니다.
- 새 참고 자료가 판단 품질을 높이지 않고 일상 prompt context만 시끄럽게 만듭니다.
