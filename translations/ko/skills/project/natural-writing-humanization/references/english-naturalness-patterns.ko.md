# 영어 자연스러움 점검

영어 README, 문서, 제품 노트, PR 설명, 배포 노트, 인수인계 문서를 다듬을 때 사용합니다.

## 먼저 볼 것

- 주어와 동사를 가깝게 둡니다. 넓은 도입문보다 실제 행동을 먼저 씁니다.
- 막연한 칭찬은 실제 동작, 한계, 근거로 바꿉니다.
- 문장 길이는 필요할 때만 조절합니다. 사람처럼 보이려고 일부러 극적인 리듬을 만들 필요는 없습니다.
- 기술 독자는 보통 낙관적인 포장보다 정확한 주의사항을 더 선호합니다.
- 명령어, 파일 경로, 설정 키, API 필드, 버전, 제품명은 그대로 보존합니다.

## AI식 영어 신호

- `in today's fast-paced landscape`, `the realm of`, `delve into`, `it is important to note` 같은 넓은 도입문.
- `pivotal`, `crucial`, `transformative`, `game-changing`, `seamless`, `robust`, `comprehensive`, `powerful` 같은 과한 가치 단어.
- `not only ... but also ...`처럼 균형을 맞추려는 문장 구조가 반복되는 경우.
- `, ensuring`, `, highlighting`, `, underscoring`, `, allowing`, `, enabling`처럼 쉼표 뒤 분사구가 의미를 늘리는 경우.
- 실무 문서인데 긴 대시, 수사적 대비, 연출된 전환이 과하게 들어간 경우.

## 고치는 방법

- “This powerful tool enables users to...”는 “Use this command to...”나 “The command reports...”처럼 직접 씁니다.
- 추상적인 장점은 확인 가능한 결과로 바꿉니다. 예: “improves reliability”보다 “fails when a path leaves the project root”.
- 진짜 주장 앞의 준비 문장을 지웁니다.
- 늘어진 문장은 결정 하나와 주의사항 하나로 나눕니다.
- 휴리스틱 검사라면 “proves”보다 “reports”, “checks”, “suggests”, “flags”를 씁니다.

## 검토 질문

- 형용사를 지워도 문장이 여전히 사실인가?
- 문단이 독자에게 결정, 명령, 주의사항, 근거 중 무엇을 주는가?
- 한계와 주의사항이 행동하기 충분히 보이는가?
- 기술 식별자가 모두 정확히 보존되었는가?
