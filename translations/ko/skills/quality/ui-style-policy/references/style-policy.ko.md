# UI 스타일 정책 호환 참고 자료

기본 라우트: `frontend/style-policy-selection`.

저장소 근거가 뒷받침하는 가장 좁은 정책을 사용합니다.

## 결정 순서

1. **디자인 시스템 우선**
   - 공유 컴포넌트, 토큰, 변형, 슬롯, UI 기본 요소가 있습니다.
   - 직접 CSS, 유틸리티 클래스, 인라인 스타일을 쓰기 전에 먼저 재사용합니다.

2. **CSS/클래스 우선**
   - 스타일시트, CSS 모듈, 스코프 CSS, 의미 있는 클래스명이 프로젝트 관례입니다.
   - 재사용 레이아웃, 변형, 가상 선택자, 미디어 쿼리, 컨테이너 쿼리는 CSS/클래스에 둡니다.

3. **유틸리티 클래스 우선**
   - Tailwind 방식의 유틸리티나 원자적 클래스 조합이 프로젝트 관례입니다.
   - 새 CSS를 추가하기 전에 설정된 토큰과 변형을 조합합니다.

4. **인라인 스타일 우선**
   - 프로젝트 문서, 코드, 사용자 지시가 컴포넌트 로컬 인라인 스타일 객체를 명시적으로 선호합니다.
   - 인라인 스타일은 컴포넌트 로컬, 동적 값, 상태에서 파생되는 값으로 제한합니다.

## 충돌 처리

- 최신 사용자 지시와 저장소 문서는 일반 스타일 선호보다 우선합니다.
- 기존 로컬 컴포넌트 패턴은 외부 스타일 조언보다 우선합니다.
- 필요한 상태를 디자인 시스템 기본 요소가 지원하면 일회성 커스텀 스타일보다 우선합니다.
- 프로젝트가 여러 방식을 섞어 쓰면 문서화된 이전 계획이 없는 한 수정하는 컴포넌트가 이미 쓰는 방식을 따릅니다.

## 문서화

정책이 나중에도 중요하다면 다음을 기록합니다.

- 선택한 정책
- 근거 출처
- 허용되는 예외
- 눈에 보이는 UI 변경의 검증 기대치
