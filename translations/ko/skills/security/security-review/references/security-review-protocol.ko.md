# 보안 검토 프로토콜

넓은 보안 검토를 구체적인 증거와 결정이 있는 개발 점검으로 좁힐 때 이 참고 자료를 사용합니다.

## 검토 범위 정하기

- 바뀐 표면을 적습니다: 라우트, 컴포넌트, 서비스, 작업, 권한 규칙, 의존성, 설정, 스키마, 저장 경로, 연동, 에이전트/도구 표면.
- 보호 자산을 식별합니다: 자격 정보, 세션, 토큰, 사용자 데이터, 업무 데이터, 감사 데이터, 파일, 모델 메모리, 로그, 실행 권한.
- 행위자를 식별합니다: 익명 사용자, 인증 사용자, 관리자, 내부 운영자, 서비스 계정, worker, 웹훅 발신자, MCP 서버, 의존성, 공격자가 조작한 콘텐츠.
- 신뢰 경계를 그립니다: 브라우저/서버, 공개/비공개 네트워크, 테넌트/관리자, 프로젝트/전역 설정, 도구/모델, 로컬/원격 파일시스템, 내부/외부 서비스.
- 진입점과 sink를 나열합니다: 입력, 변환, 저장, 명령 실행, 네트워크 호출, 외부 메시지, 로그, 알림, 내보내기, 캐시.

## 위험 분류

| 분류 | 검토 초점 |
|---|---|
| 비밀값 노출 | 환경 변수, API key, 토큰, 로그, 브라우저 번들, 빌드 산출물, 충돌 보고서, 복사된 fixture. |
| 인증/인가 | 누락된 신원 확인, confused deputy 경로, 역할 드리프트, 테넌트 경계 누출, 오래된 세션 처리. |
| 입력/출력 | 주입, 안전하지 않은 파싱, 파일 경로 순회, XSS, SSRF, 역직렬화, 안전하지 않은 템플릿 출력, 콘텐츠 타입 불일치. |
| 데이터 무결성 | 무단 변경, 재전송, 중복 전달, 오래된 쓰기, 약한 검증, 감사 기록 누락. |
| 공급망 | 새 의존성, 설치 스크립트, 바이너리 산출물, 전이 CVE, 라이선스/고지 드리프트, 신뢰할 수 없는 생성 코드. |
| 운영 | 디버그 endpoint, 과도하게 자세한 오류, 안전하지 않은 기본값, rate limit 누락, 안전하지 않은 rollback, 모니터링 사각지대. |

## 검토 절차

1. 파일 diff만 보지 말고 데이터 흐름에서 시작합니다. 민감 값이 원천에서 sink까지 어떻게 이동하는지 따라갑니다.
2. 모든 우려를 공격 이야기로 바꿉니다: 행위자, 전제 조건, 행동, 영향, 관측 가능한 신호.
3. 경로를 닫는 가장 작은 완화책을 우선합니다: 기본 거부, 신뢰 경계 전 검증, 원천 단계 마스킹, 범위 제한, 명시 승인.
4. 공격 경로가 안정적으로 재현될 수 있으면 회귀 검사를 추가합니다.
5. 완전히 닫을 수 없는 위험은 남은 위험, 소유자, 모니터링 신호, rollback 또는 격리 계획을 기록합니다.

## 증거

- 진입점, 정책 검사, 검증, sink, 테스트의 코드 위치.
- 의존성, 라이선스, 정적 분석, 테스트 검사를 실행했다면 정확한 명령 출력.
- 계정/역할, 라우트/작업, payload 형태, 예상 거부 또는 마스킹 동작을 포함한 수동 검증 노트.
- 위험 결정: 수용, 완화, 소유자를 둔 연기, 차단.

## 중단 조건

- 비밀값이 source control, 로그, 생성 산출물, 브라우저에서 보이는 payload, 외부 서비스에 기록될 수 있습니다.
- 변경이 명확한 정책 검사 없이 테넌트, 역할, 관리자, 실행, 네트워크 신뢰 경계를 넘습니다.
- 의존성이나 도구가 필요한 범위를 넘는 파일시스템, 셸, 네트워크, 토큰, 프로젝트 설정 접근을 요구합니다.
- 민감 데이터 경로나 실행 경계를 식별할 수 없습니다.
- 새 공개 진입점에 rate limit, auth policy, input validation, failure behavior가 없습니다.
