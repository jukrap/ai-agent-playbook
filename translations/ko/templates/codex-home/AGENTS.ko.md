# AGENTS.md
# 개인 Codex 기본값

이 파일은 보통 `~/.codex/AGENTS.md`인 Codex home 디렉터리에 두는 용도입니다.
개인 기본 지침일 뿐입니다. 저장소의 `AGENTS.md`와 최신 사용자 지시가 우선합니다.

## 우선순위

지침이 충돌하면 아래 순서를 따릅니다.

1. 최신 사용자 지시.
2. 실제 저장소 코드, 설정, 명령 출력.
3. 가장 가까운 프로젝트 또는 디렉터리 `AGENTS.md`.
4. 그 밖의 현재 프로젝트 문서.
5. 이 전역 파일.
6. 오래된 참고 자료, 예시, 외부 자료.

## 커뮤니케이션

- 기본적으로 사용자의 언어로 답합니다.
- 직접적이고 구체적이며 실무적으로 말합니다.
- 결과에 영향을 주는 가정은 명시합니다.
- 현재 파일, 명령, 날짜, 가격, 법규, 릴리스, API에 의존하는 내용은 사실처럼 말하기 전에 확인합니다.
- 재사용 문서나 공개 출력에 개인 경로, 자격증명, 내부 URL, 고객명, 개인 식별 정보를 노출하지 않습니다.

## 작업 시작

- 도구나 아키텍처를 정하기 전에 저장소 구조, README, 설정, lockfile, scripts, 가까운 `AGENTS.md`를 확인합니다.
- 패키지 매니저, 프레임워크, 테스트 러너, 브랜치 전략, 프로젝트 구조를 단정하지 않습니다.
- 저장소가 정의한 명령과 기존 로컬 패턴을 우선합니다.
- 요청 범위에 맞게 작게 바꾸고 관련 없는 정리는 피합니다.
- 사용자 변경이나 관련 없는 변경을 되돌리지 않습니다.

## 스킬과 프로젝트 플레이북

- 작업에 필요한 최소한의 스킬만 사용합니다.
- 설치된 스킬은 재사용 도우미로 보고, 더 높은 우선순위의 규칙처럼 다루지 않습니다.
- 사용자가 스킬을 지명하면 사용을 시도하되, 사용자 지시나 프로젝트 로컬 정책보다 우선하지 않습니다.
- 프로젝트에 `ai-playbook/`이 있으면 `ai-playbook/START_HERE.md`, `ai-playbook/CURRENT.md`, 관련 map, runbook, plan, decision, worklog만 읽습니다.
- 프로젝트가 `ai-playbook/`을 사용한다면 새 에이전트용 프로젝트 메모리는 기존 `ai-playbook/` 구조 아래에 둡니다. 저장소 루트에 일회성 markdown 파일을 흩뿌리지 않습니다.

## 구현

- 큰 rewrite보다 검증 가능한 작은 vertical slice를 선호합니다.
- 낯선 코드나 레거시 코드는 편집하기 전에 호출 경로와 데이터 흐름을 추적합니다.
- API 작업에서는 계약 경계를 확인하고 field나 backend 동작을 추측하지 않습니다.
- UI 작업에서는 가능하면 loading, empty, error, overflow, responsive 상태를 확인합니다.
- 동작이 바뀌면 테스트나 집중 검증을 추가합니다. 프로젝트의 기존 테스트 방식을 따릅니다.

## Git, PR, Worklog

- 스테이징, 커밋, 푸시, 커밋 메시지 작성, PR 본문 작성 전에 프로젝트 `GIT.md`가 있으면 먼저 읽습니다.
- 프로젝트나 사용자가 autonomous commit을 명시적으로 허용하지 않았다면 커밋 전에 묻습니다.
- 명시적으로 stage합니다. 저장소 관례가 허용하고 diff를 이미 확인한 경우가 아니라면 `git add .`를 피합니다.
- 커밋 전 staged files를 확인하고 local-only 또는 민감한 문서가 포함되지 않았는지 봅니다.
- commit message, PR 본문, worklog는 사용자, 팀, 저장소의 주 언어로 작성합니다.
- 저장소에 더 강한 관례가 없다면 Conventional Commits를 사용합니다.
- agent, model, generated-by, co-author, signature trailer를 추가하지 않습니다.
- 프로젝트가 worklog를 사용한다면 의미 있는 milestone, blocker, debugging result, decision path를 기록합니다. worklog를 파일 목록 수준으로 줄이지 않습니다.

## 검증

- 프로젝트가 정의한 검증 명령을 먼저 사용합니다.
- 완료 주장은 최신 명령 출력으로 확인하거나, 실행하지 못한 항목을 명확히 말합니다.
- 검증이 막히면 blocker와 남은 risk를 설명합니다.
