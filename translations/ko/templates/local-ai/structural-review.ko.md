# 구조 리뷰

에이전트가 저장소 구조, 정리 후보, 중복, 죽은 코드, 모듈 크기, 의존성 순환, 리팩터링 우선순위를 검토할 때 이 가이드를 사용합니다.

## 근거 규칙

- 현재 저장소에서 나온 근거 없이 구조 주장을 하지 않습니다.
- 스캔 범위를 밝히지 않고 “없다”고 단정하지 않습니다.
- 최신성을 확인하지 않은 오래된 감사 산출물을 사용하지 않습니다.
- 정적 분석은 근거일 뿐 최종 아키텍처 판정이 아닙니다.
- 근거가 불완전하면 글로 빈틈을 메우지 말고 무엇을 모르는지 말합니다.

예:

- 좋음: `현재 전체 감사에서 스캔한 216개 파일 기준 순환 의존성 0건`.
- 나쁨: `순환 의존성이 없습니다`.
- 좋음: `이 helper는 정리 후보입니다. 스캔한 그래프에서 consumer를 찾지 못했습니다`.
- 나쁨: `이 helper는 확실히 사용되지 않습니다`.

## 리뷰 주기

- 첫 점검, 오래된 기준선, 큰 리팩터링, 실사 성격의 검토, 리팩터링 후 확인에는 전체 감사를 사용합니다.
- 최신 기준선이 있는 뒤의 작은 국소 질문에는 빠른 후속 확인을 사용합니다.
- 일반 대화에서는 모든 산출물을 나열하지 말고 상위 1~3개 발견 사항을 요약합니다.
- 공식 리뷰에서는 스캔 범위, 신뢰 한계, 필요한 정확한 파일 또는 줄을 포함합니다.

## 선택적 도구

프로젝트가 저장소 감사 또는 구조 검증 도구를 설치했다면 기술 스택에 맞을 때 기계적으로 확인한 근거로 사용합니다.

- 다른 에이전트 런타임의 슬래시 명령 문법을 Codex에 그대로 복사하지 않습니다.
- 프로젝트가 선택하지 않았다면 작성 전후 게이트가 활성화되어 있다고 가정하지 않습니다.
- 일반 대화 사용자에게 기계용 의도 JSON을 직접 쓰라고 요구하지 않습니다.
- 프로젝트가 명시적으로 추적하지 않는 한 생성된 감사 산출물은 로컬에 둡니다.

## 리팩터링 계획

기본적으로 작은 다음 단위 하나를 선호합니다.

- 담당 파일, 모듈, 도메인 영역을 이름으로 지정합니다.
- 범위 밖에 둘 것을 명시합니다.
- 분리하면 저장소가 불일치하게 남을 때는 영향을 받을 가능성이 큰 파일이나 consumer를 포함합니다.
- 검증 명령 또는 감사 확인을 포함합니다.
- `clean architecture`, `fix all dead code` 같은 넓은 목표를 피합니다.

## 출력 형태

간결한 리뷰에는 아래 구조를 사용합니다.

```md
## 현재 상태
- 이미 안정적이거나 스캔이 뒷받침할 수 있는 내용.

## 다음 정리 후보
- 근거가 있는 가장 작은 개선과 시작 위치.

## 지금은 유지
- 보호 대상, 낮은 신뢰도, 생성 산출물, 공개 API, 관례가 정한 표면.

## 신뢰도
- 스캔 범위, 최신성, 모르는 것, 다음 확인.
```

코드 리뷰 발견 사항은 스타일이나 정리 제안보다 버그, 회귀, 위험을 먼저 제시합니다.
